## 一、4T 测评机制要点（By May）

> **核心前提：4T 成绩 = 省赛/国赛成绩，务必针对 4T 做题。**

### 1. 数码管闪烁
- 4T **不做动态刷新检测**，直接读寄存器保持显示。
- 闪烁对 4T 无影响，读取 DS18B20、DS1302、超声波等模块时，**可以大胆关闭中断（EA=0）**。

### 2. LED 闪烁周期
- 4T 约用 **5 秒** 检测闪烁周期，误差窗口很小（100ms 周期只允许 ±10ms）。
- **根本原因**：闪烁逻辑写在普通函数里，被中断打断导致响应不及时。
- **解决方案：LED 闪烁逻辑必须写在定时器中断里。**

```c
// 错误做法：闪烁逻辑在普通函数中
void led_proc() {
    if(uwTick % 100 == 0) {
        ucled[0] = !ucled[0]; // 容易被中断打断，周期不准
    }
    led_disp(ucled);
}

// 正确做法：闪烁逻辑写入中断
void Timer1_Isr(void) interrupt 3 {
    uwTick++;
    // 闪烁逻辑在中断中，保证实时性
    if(led_flash_enable) {
        if(++led_cnt >= 100) {
            led_cnt = 0;
            ucled[0] = !ucled[0];
            led_disp(ucled); // 或只维护开关变量，显示在外部
        }
    }
}
```

### 3. NE555 频率采样
- **推荐采样周期 200ms**，而非 1 秒。
- 1 秒采样会导致频率突变时数据严重滞后，造成错误扣分。

### 4. AD 双通道读取（PCF8591）
- 因转换需要时间，读到的是**上次转换结果**，双通道同时读时两路数值会互换。
- **解决方案：两个通道都读两遍（读两边）**，取第二次结果。

### 5. DS18B20 温度
- 4T 是虚拟环境，**不存在 750ms 转换时间，也不存在上电 85°C 默认值**。
- **不需要**上电延时 750ms、空读一次、判断 85°C，否则可能扣分。
- 跳变问题：开关中断（EA=0 / EA=1）解决。

### 6. 超声波距离偏差
- 4T 与实际结果可能差 3，4T 允许误差 ±3。
- **推荐：结果手动 +2**，无论 bug 修没修，+2 是最稳的。

### 7. EEPROM "上锁"问题
| 场景 | 处理方式 |
|------|----------|
| **指定存储地址** | **不加校验**（4T 预留了模板参数，加校验会跳过读取导致变成 0） |
| **不指定存储地址** | **加校验**（判断是否是自己写过的 EEPROM） |

### 8. 串口
- 底层简单，难点在逻辑层，**省赛必考**，务必掌握。

### 9. 推荐代码架构

```
主循环
├── key_proc()       // 按键处理
├── uart_proc()      // 串口处理
└── seg_proc()       // 核心主函数（100ms 调用）
      ├── 数据读取（超声波/温度/AD/时钟 一次性全读，约 ≤20ms）
      ├── 数据处理
      ├── 数据显示
      └── LED 逻辑处理（维护状态变量，开关由中断执行）

定时器中断（1ms）
└── LED 闪烁开关逻辑
```

### Bug 1【严重】ultrasound.c — for 循环参数顺序写反

**问题**：`Ultra_Data_Init()` 中 for 循环三个参数顺序错误，导致循环体一次都不执行，TX 引脚从未翻转，超声波完全收不到触发信号。

```c
// ❌ 错误代码（ultrasound.c 第 23 行）
for(i=0; i++; i<8)
// 解释：for(初始; 条件; 迭代)
//       条件 = i++，初始 i=0，后置++ 返回 0（false），循环一次都不跑！

// ✅ 正确代码
for(i=0; i<8; i++)
```

**完整正确的 Ultra_Data_Init：**
```c
void Ultra_Data_Init() {
    unsigned char i = 0;
    EA = 0;
    for(i = 0; i < 8; i++) {   // ← 修正参数顺序
        US_TX = 1;
        Delay12us();
        US_TX = 0;
        Delay12us();
    }
    EA = 1;
}
```

### Bug 2【严重】ultrasound.c — 缺少等待回波上升沿

**问题**：发完触发脉冲后，US_RX 初始是低电平。代码直接开定时器并进入 `while((CF==0)&&(US_RX==1))`，由于 `US_RX==1` 为 false，while 立刻退出，测距结果永远接近 0 或返回 999。

```c
// ❌ 错误代码（ultrasound.c 第 49~54 行）
Ultra_Data_Init();
CMOD = 0X00;
CH = CL = 0;
CR = 1;
while((CF==0) && (US_RX==1));   // US_RX 初始是低，一进来就立刻退出！
CR = 0;

// ✅ 正确代码（增加等待回波上升沿）
Ultra_Data_Init();
CMOD = 0X00;
CH = CL = 0;
while(!US_RX);                  // ← 新增：等待回波信号变高（echo 开始）
CR = 1;
while((CF==0) && (US_RX==1));   // 等待回波下降或定时器溢出
CR = 0;
```

**超声波完整正确函数：**
```c
unsigned int Ultra_Data_Wave() {
    unsigned long int time = 0;
    Ultra_Data_Init();
    CMOD = 0X00;
    CH = CL = 0;
    while(!US_RX);                       // 等待回波上升沿
    CR = 1;
    while((CF==0) && (US_RX==1));        // 等待回波下降或超时
    CR = 0;
    if(CF == 0) {
        time = CH << 8 | CL;
        return (unsigned int)(time * 17 / 1000) + 2;  // 整数运算，+2 补偿
    } else {
        CF = 0;
        return 999;
    }
}
```

### Bug 3【严重】led.c — Beep 函数忘记给 P0 赋值就开锁存器

**问题**：蜂鸣器控制数据存在 `temp_2` 中，但在打开锁存器（P2[7:5]=101）之前，**从未将 `temp_2` 写入 P0**。锁存进去的是数码管刷新留在 P0 上的随机值，蜂鸣器控制完全失效。

对比 `led_disp` 的正确写法：先写 P0，再开锁存。

```c
// ❌ 错误代码（led.c Beep 函数）
if(temp_2 != temp_2_old) {
    // 这里漏掉了 P0 = temp_2 ！
    temp = P2 & 0x1f;
    temp = temp | 0xa0;     // 打开蜂鸣器锁存器
    P2 = temp;
    temp = P2 & 0x1f;
    P2 = temp;
    temp_2_old = temp_2;
}

// ✅ 正确代码
if(temp_2 != temp_2_old) {
    P0 = temp_2;            // ← 必须先把控制数据写入 P0
    temp = P2 & 0x1f;
    temp = temp | 0xa0;     // 再打开蜂鸣器锁存器
    P2 = temp;
    temp = P2 & 0x1f;
    P2 = temp;
    temp_2_old = temp_2;
}
```

### Bug 4【一般】seg.c — 段码数组缺少类型声明

**问题**：`seg_dula` 数组未指定 `unsigned char` 类型，Keil C51 默认当 `int` 处理，每个元素占 2 字节，11 个元素多占 11 字节 RAM，在 RAM 紧张时可能引发问题。

```c
// ❌ 错误代码（seg.c 第 2 行）
seg_dula[] = {0xc0, 0xf9, 0xa4, 0xb0, 0x99, 0x92, 0x82, 0xf8, 0x80, 0x90, 0xff};

// ✅ 正确代码
unsigned char seg_dula[] = {0xc0, 0xf9, 0xa4, 0xb0, 0x99, 0x92, 0x82, 0xf8, 0x80, 0x90, 0xff};
```

### Bug 5【一般】iic.c — EEPROM 写后延时放在 I2CStop() 之前

**问题**：AT24C02 的内部写周期（约 5ms）是在收到 STOP 信号后才开始的。当前代码将延时放在 `I2CStop()` 之前，等于在 EEPROM 开始写之前就在延时，Stop 之后如果立刻读取会 NACK。

```c
// ❌ 错误代码（iic.c 第 190~194 行）
// 延时 → 再 Stop
I2C_Delay(250); I2C_Delay(250); /* ... 共10次 */
I2CStop();   // Stop 之后 EEPROM 才开始内部写，延时白等了

// ✅ 正确代码
I2CStop();   // 先发 Stop，EEPROM 开始内部写周期
// Stop 之后再等待 5ms
I2C_Delay(250); I2C_Delay(250); /* ... 共10次 */
```

---

### Bug 6【轻微】ultrasound.c — 浮点运算拖慢速度

**问题**：8051 没有浮点运算单元，`time * 0.017` 会调用软件浮点库，运算很慢。

```c
// ❌ 较慢（浮点）
return time * 0.017 + 2;

// ✅ 更快（整数运算，结果相同）
return (unsigned int)(time * 17 / 1000) + 2;
```
