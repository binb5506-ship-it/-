## 一、你原代码里的主要错误

### 1. “接近”判定方向写反了

题目要求：
- 距离值低于距离参数 `PL` 时，触发“接近”判定。

你原来的写法：

```c

{
    Distance_Flag = 1;
}
```

正确写法：

```c
Near_Flag = (Distance < Dist_Param) ? 1 : 0;
```

解释：
- 题目要求是“低于参数”才算接近，所以一定是 `<`，不是 `>`。
- 你原来的逻辑会变成“距离越远，越容易判定接近”，与题目完全相反。
- 我在新代码 `Distance_Proc()` 里已经改成正确判定。

### 2. 接近标志和高温标志只会置 1，不会恢复 0

你原来的写法本质上是这种模式：

```c
if(Wendu > Wendu_Can)
{
    Wendu_Flag = 1;
}

if(Distance > Distance_Can)
{
    Distance_Flag = 1;
}
```

正确写法：

```c
Temp_Flag = (Temperature > Temp_Param) ? 1 : 0;
Near_Flag = (Distance < Dist_Param) ? 1 : 0;
```

解释：
- 实时系统里的状态标志不能只会“触发”，还必须会“恢复”。
- 否则一旦某次超限，以后即使条件已经消失，标志位仍然是旧状态。
- 这会直接导致继电器无法断开，LED 状态也会错误。

### 3. `L1~L4` 没有按题目要求受“接近判定”控制

题目要求：
- 只有在触发“接近”判定时，`L1~L4` 才按光强等级点亮。
- 没有接近时，`L1~L4` 必须全部熄灭。

你原来的写法：

```c

else if(Dengji==2) { ucled[0]=1; ucled[1]=1; }
else if(Dengji==3) { ucled[0]=1; ucled[1]=1; ucled[2]=1; }
else if(Dengji==4) { ucled[0]=1; ucled[1]=1; ucled[2]=1; ucled[3]=1; }
```

正确写法：

```c
if(Near_Flag)
{
    if(Light_Level >= 1) led_buf[0] = 1;
    if(Light_Level >= 2) led_buf[1] = 1;
    if(Light_Level >= 3) led_buf[2] = 1;
    if(Light_Level >= 4) led_buf[3] = 1;
}
```

解释：
- 你原来的写法只看了光强等级，没有先判断“是否接近”。
- 所以会出现“远处也亮 L1~L4”的问题。
- 我把这部分封装到了 `Build_Live_Led_Buffer()` 里，逻辑更清楚。

### 4. `Get_Da()` 没有进入调度器，而且 ADC 换算也不够准确

第一，函数写了但没调度：

```c

    ...
}
```

但任务表里原本没有它。

第二，换算写法是：


正确写法：

```c

else if(adc_val >= 102) Light_Level = 2;
else if(adc_val >= 26)  Light_Level = 3;
```

解释：

- 我改成了“直接用 ADC 原始值分段”，既准，也更容易读懂。

### 5. 温度、距离、光强界面混用了“旧值快照”和“实时值”

### 6. 距离检测在同一次任务里连续读取了两次


- 我改成每次调度只读一次，把上一秒距离存在 `Prev_Distance`，严格按题目做。

### 7. 运动状态的 3 秒锁定机制缺失

题目要求：
- 当运动状态变化时，立即更新并锁定 3 秒。
- 锁定期间继续采样距离，但不允许再改变运动状态。

你原来的代码没有这个机制。

正确写法：

```c
if(Motion_Lock)
{

    {
        Prev_Distance = Distance;
        return;
    }
    Motion_Lock = 0;
}

if(new_motion != Motion_State)
{
    Motion_State = new_motion;
    Motion_Lock = 1;
    Motion_Lock_Start = uwTick;
}
```

解释：
- 这是本题最重要的状态机之一。
- 只要缺失这个锁定，`L8` 的状态就会频繁变化，不符合题目示意图。

### 8. 参数界面没有真正锁定 LED 和继电器输出

题目要求：
- 进入参数界面时，LED 和继电器状态锁定。
- 退出参数界面后，恢复实时输出。

你原来的代码只是在显示上保存了一些旧值，但没有真正锁定“实际输出”。

正确写法：

```c
void Capture_Output_Lock(void)
{
    unsigned char i;

    for(i = 0; i < 8; i++)
    {
        Lock_Led_Buf[i] = Led_Buf[i];
    }
    Lock_Relay_State = Relay_State;
}
```

以及：

```c
if(Page_Mode == PAGE_PARAM)
{
    Led_Disp(Lock_Led_Buf);
}
else
{
    Build_Live_Led_Buffer(Led_Buf);
    Led_Disp(Led_Buf);
}
```

解释：
- 题目说的是“状态锁定（不可变化）”，不是“重新按锁存的逻辑值计算”。
- 所以我锁住的是“进入参数界面瞬间的实际 LED 输出结果”和“继电器当前状态”。
- 这样即使 `L8` 进入参数界面前正在闪烁，也会冻结在那一瞬间，符合“不可变化”。

### 12. 按键边沿提取公式不适合你当前的键值返回方式


这种写法适用于“位图式按键值”。

```c

    Key_Down = Key_Val;

else

    Key_Down = 0;

    Key_Up = Key_Old;
}

}
```

解释：
- 你这个驱动不是按位返回按键状态，所以不能继续套“位运算边沿公式”。
- 我改成了更适合当前驱动的“从无到有 / 从有到无”边沿判断。

- 如果 1ms 时基本身不准，后面所有“周期任务”都会跟着漂。
- 我把它改成了标准的 `12MHz / 12T / 1ms` 配置。

## 四、你的代码缺点和后续建议

### 1. 容易只写“触发逻辑”，忘记写“恢复逻辑”

这是你这份代码里最典型的问题。

表现为：
- 标志位只置 1，不清 0
- 继电器只考虑吸合，不完整管理实际状态
- 进入参数界面只想着“显示切换”，没把“退出后的恢复”整体想清楚

建议：
- 以后写每一个状态变量时，都问自己一句：
  “它什么时候置位？什么时候清零？什么时候保持不变？”

### 2. 容易把“题目要求的状态机”写成“几个 if 拼起来”

比如这题的运动状态，真正要求的是：
- 每 1 秒采样
- 比较相邻两次距离
- 状态变化后锁定 3 秒

这已经不是一个简单的 `if-else` 了，而是完整状态机。

建议：
- 碰到“变化后保持一段时间”“某界面进入后冻结输出”“长按 2 秒触发”这种题目描述时，第一反应就应该是：
  “我要不要单独加状态变量和时间戳？”

### 3. 对驱动返回值的理解还不够严谨

比如：
- `Key_Read()` 返回的是“键值编号”，不是位图
- `Ut_Wave_Data()` 返回 `0` 可能意味着超时/无效，不一定是真正 0cm

建议：
- 写上层逻辑前，先弄清楚驱动到底返回什么格式。
- 不要先套自己熟悉的模板公式，要先看接口本身。

### 4. 题目细节容易漏掉“边界条件”

例如：
- 参数上限下限
- 统计界面长按 2 秒
- 参数界面默认回到 `PC`
- 参数界面输出锁定

建议：
- 以后做这类比赛题，先把题目里的“限定词”单独抄出来：
  - “仅当”
  - “同时”
  - “保持 2 秒以上”
  - “默认”
  - “锁定 3 秒”
  - “范围 20~80”

这些词基本就是得分点。
