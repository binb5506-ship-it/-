# 今日错误清单（2026-04-08）

---

## 错误1：声明末尾缺分号

**错误写法**
```c
pdata unsigned char ucled[]={1,0,1,0,1,0,1,0}
```
**正确写法**
```c
pdata unsigned char ucled[]={1,0,1,0,1,0,1,0};
```
**解释**  
C 语言每条语句必须以分号结尾。缺少分号后编译器会把下一行也当成当前语句的一部分，导致后面的 `void main()` 等语法全部报错，报错信息看起来很多，但根源只是这一个分号。

## 错误2：变量名大小写不一致

**错误写法**
```c
unsigned char Temp_1_Old = 0xff;   // 声明用大写 T

if(temp_1 != temp_1_Old)           // 使用时小写 t ❌
    temp_1_Old = temp_1;           // 使用时小写 t ❌
```
**正确写法**
```c
if(temp_1 != Temp_1_Old)           // 与声明一致
    Temp_1_Old = temp_1;
```
**解释**  
C 语言严格区分大小写，`Temp_1_Old` 和 `temp_1_Old` 是两个完全不同的变量。使用时拼写与声明不一致，编译器会报 `undefined identifier`。

## 错误3：头文件路径错误

**错误写法**
```c
#include "led.h"     // led.h 在 Driver 文件夹，找不到
```
**正确写法**
```c
#include "../Driver/led.h"   // 用相对路径
```
**解释**  
`main.c` 在 `User/` 目录，`led.h` 在 `Driver/` 目录，两者不在同一层。用 `"led.h"` 编译器只会在当前目录找，找不到就报 `can't open file`。用 `../Driver/led.h` 或在 Keil 的 `Options for Target → C51 → Include Paths` 里加上 `..\Driver` 可解决。

## 错误4：变量声明写了两个类型

**错误写法**
```c
idata unsigned char task_t Scheduler_Task[] = { ... };
//       ↑ 多余，与 task_t 冲突
```
**正确写法**
```c
idata task_t Scheduler_Task[] = { ... };
```
**解释**  
一个变量只能有一个类型。`unsigned char` 和 `task_t` 同时出现，编译器在读完 `unsigned char` 后认为声明已结束，看到 `task_t` 就不认识了，报 `missing ';' before 'Scheduler_Task'`。遇到这个报错要往前一行找原因。

## 错误5：结构体成员名写错

**错误写法**
```c
// 结构体定义的字段名是 last_ms、rate_ms
if(now_time >= Scheduler_Task[i].last.ms + Scheduler_Task[i].rate.ms)
    Scheduler_Task[i].last.ms = now_time;
```
**正确写法**
```c
if(now_time >= Scheduler_Task[i].last_ms + Scheduler_Task[i].rate_ms)
    Scheduler_Task[i].last_ms = now_time;
```
**解释**  
结构体字段名是 `last_ms`，不是嵌套结构体 `last.ms`。写成 `last.ms` 意味着 `last` 本身也是一个有 `ms` 字段的结构体，与定义不符，编译报错。

## 错误6：main() 里没有调用 Timer1_Init()

**错误写法**
```c
void main()
{
    Scheduler_Init();
    while(1) { Scheduler_Run(); }
    // 定时器从未启动
}
```
**正确写法**
```c
void main()
{
    Scheduler_Init();
    Timer1_Init();   // 必须调用，否则定时器不工作
    while(1) { Scheduler_Run(); }
}
```
**解释**  
定时器不会自动启动，必须手动初始化。不调用 `Timer1_Init()`，中断永远不触发，数码管扫描不执行，`uwTick` 不递增，调度器也无法计时。

## 错误7：调度器时间来源未初始化

**错误写法**
```c
void Scheduler_Run()
{
    unsigned long int now_time;   // 未赋值，值是随机垃圾
    ...
}
```
**正确写法**
```c
idata unsigned long int uwTick = 0;   // 全局计数器

void Isr_Timer1() interrupt 3
{
    uwTick++;   // 每 1ms 加一次
    ...
}

void Scheduler_Run()
{
    unsigned long int now_time = uwTick;   // 读全局时间
    ...
}
```
**解释**  
调度器需要知道"现在是第几毫秒"才能判断任务是否到期。时间必须由定时器中断每 1ms 递增一次，`Scheduler_Run` 读这个全局变量。用未初始化的局部变量，值是随机的，调度逻辑完全失效。

## 错误8：L15 警告——同一函数被主程序和中断同时调用

**错误写法**
```c
// 主程序路径：Scheduler_Run → Seg_Proc → Seg_Disp
void Seg_Proc() { Seg_Disp(1,2,1); }

// 中断路径：Isr_Timer1 → Seg_Disp
void Isr_Timer1() interrupt 3 { Seg_Disp(...); }
```
**正确写法**
```c
// 主程序只写缓冲区
void Seg_Proc() { Seg_Buf[1] = 2; }

// 中断负责读缓冲区并驱动显示
void Isr_Timer1() interrupt 3 { Seg_Disp(Seg_Pos, Seg_Buf[Seg_Pos], 0); }
```
**解释**  
C51 编译器为节省 RAM，让多个函数共享同一块内存存局部变量。若主程序和中断同时调用同一函数，中断会覆盖主程序的局部变量，导致数据错乱。正确做法是：中断只负责驱动底层硬件，主程序只负责更新数据缓冲区，两者分工明确。

## 错误9：seg.c 缺少段码锁存操作

**错误写法**
```c
P0 = seg_dula[dula];   // 段码写到 P0
if(point) P0 &= 0x7f;
// 没有锁存操作 → 数据进不了锁存器，数码管始终灭
```
**正确写法**
```c
P0 = seg_dula[dula];
if(point) P0 &= 0x7f;
// 再触发一次 Y7 锁存
temp = P2 & 0x1f;
temp = temp | 0xe0;
P2 = temp;
temp = P2 & 0x1f;
P2 = temp;
```
**解释**  
板子上用 74HC573 锁存器驱动数码管，数据必须先送到 P0，再"开锁存→关锁存"才能被锁进去。函数里对"消隐"和"位选"都做了锁存，唯独最后的段码数据忘记锁存，导致段锁存器里一直是第一步写入的 `0xff`（全灭）。

## 错误10：头文件缺少函数声明

**错误写法**
```c
// key.h
#include <STC15F2K60S2.H>
// 没有 Key_Read 的声明
```
**正确写法**
```c
// key.h
#include <STC15F2K60S2.H>
unsigned char Key_Read();
```
**解释**  
头文件的作用就是让其他 `.c` 文件知道函数的存在和参数类型。没有声明，编译器不知道 `Key_Read` 返回什么类型、接受什么参数，会报 `missing function-prototype`。

---

## 错误11：全局变量用函数调用初始化

**错误写法**
```c
idata unsigned char k = Key_Read();   // ❌ 不允许
```
**正确写法**
```c
idata unsigned char k = 0;            // 全局变量只能用常量初始化

void Seg_Proc()
{
    k = Key_Read();   // 在函数里调用
    ...
}
```
**解释**  
C51（以及标准 C）规定全局变量的初始值必须是编译期可知的常量，不能是函数调用的返回值（因为程序还没运行，函数无法执行）。应声明为 `0`，在需要的时候于函数内调用 `Key_Read()` 更新。
